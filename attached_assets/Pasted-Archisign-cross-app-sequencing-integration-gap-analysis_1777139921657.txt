§B. Integration-point gaps Architrak should chew on
These are places where two apps' breakdowns describe the same wire surface but with subtle assumption-divergence risk. Better caught now than at integration-test time.

§B.1 Shared identityVerification test fixture
The 8-field block (§3.4) is constructed by Archisign (AS3), persisted by Architrak (AT4), and re-emitted by Architrak in the work-authorisation payload to Archidoc (AT4 outbound → AD3). If each app crafts its own canonical fixture, fields drift in test-only ways (e.g. one app uses signerIpAddress: "203.0.113.42", another uses "192.0.2.1"; one omits lastViewedAt). Proposal: Architrak's AT-task plan calls out a shared fixture in docs/INTER_APP_CONTRACT_v1.0.md Appendix or in a separate docs/test-fixtures.json, asserted byte-equal by all three apps.

§B.2 webhookUrl registration is one-shot per envelope (§3.7)
Archisign records webhookUrl at create-time and never re-reads it. If Architrak deploys to a new base URL between create and the envelope.signed emission (e.g. domain migration, env swap), the original webhookUrl is dead and Archisign won't fan out to the new one. Per §3.7 this is by design — Archisign does not maintain a registry. Proposal: Architrak's AT4 plan explicitly acknowledges that envelope-level webhookUrl is immutable; deployment-base-URL changes require either a finite envelope migration window or per-envelope re-create (in practice, accept the operational cost during cutover windows).

§B.3 pdfFetchUrl 60s budget vs Architrak's signed-URL TTL
§3.5.1 hard-requires Archisign-side server fetch within 60s; §3.5.1 + §AS3 budget says callers must serve URLs with TTL ≥ 60s; Architrak says 15-min by convention. Risk: Architrak ever issues a sub-60s URL (e.g. for a dev fixture) and Archisign returns 400 pdfFetchUrl unfetchable. Proposal: Architrak's AT4 plan pins "minimum signed-URL TTL = 5 minutes" as an enforced invariant in the URL-minting helper, not just by convention.

§B.4 Rate-limit budget per (key, endpoint family) — does Architrak use one key or many?
§3.6 says counters are per-(API key, endpoint family). §1.6 says Architrak's worst-case is 25 envelopes in 5 min = ~25 burst tokens consumed on create AND ~25 on send (separate buckets, fine). But: does Architrak provision one ARCHITRAK_API_KEY per environment, or per-tenant? If per-environment and Architrak grows to 5 tenants under one key, the 60 RPM ceiling is shared across them. Proposal: Architrak's AT-task plan picks a stance. Single-key-per-env is simpler and within budget today; per-tenant keys is a v1.1 scaling concern. Just call out the choice so Archisign can size the bucket store.

§B.5 /send resend-after-expired returns rotated accessToken — receiver must re-display
§3.5.4 + AS4 §3.5.2: on resend against expired, Archisign rotates the signer's accessToken, invalidates the previous, and the response includes the fresh signers. Easy mistake: Architrak's AT4 caches the original accessUrl from create-time and re-displays it after resend, leaving the architect handing out a dead link. Proposal: Architrak's AT4 plan explicitly handles "after /send resend, refresh stored accessToken and accessUrl from the response payload; never cache the create-time value past an envelope.expired event."

§B.6 §5.3.2 downstream payload shape ≠ §3.8 410 body shape
The temptation is to forward the §3.8 410 body verbatim from Archisign downstream into Architrak's outbound webhook to Archidoc. They are different shapes:

Field	§3.8 (Archisign 410 body)	§5.3.2 (Architrak → Archidoc payload)
error: "retention_breach"	✓	absent
envelopeId (Archisign's int)	✓	absent
eventId	absent	✓
eventType: "signed_pdf_retention_breach"	absent	✓
architrakDevisId	absent	✓
archisignEnvelopeId (string)	absent	✓
archidocProjectId	absent	✓
incidentRef, remediationContact, originalSignedAt, detectedAt	✓	✓
Proposal: Architrak's AT5 plan explicitly distinguishes "what Archisign returned us" from "what we tell Archidoc," and asserts the downstream payload against a fixture pinned to §5.3.2.

§B.7 Gmail X-Archidoc-Dqe-Export-Id header case sensitivity
§0.2 + §5.5 specify lower-case form per RFC 7230 §3.2 and case-insensitive read at Architrak ingestion. Risk: Archidoc's AD5 emits in Title-Case; Architrak's existing Gmail watcher reads case-sensitively. Proposal: both apps' tests pin the case-insensitive expectation explicitly. Cheap to verify, painful at integration time if missed.

§B.8 expiresAt validation floor
Contract is silent on minimum expiresAt. Archisign's AS4 stance: validates expiresAt > now() + 1 minute (rejects past dates and "essentially-now" dates as 400). No upper bound. Proposal: Architrak's AT4 plan acknowledges the 1-minute floor and picks a default (30-day was mentioned in earlier rounds — suggest pinning that as the default in Architrak's create-envelope helper).

§B.9 Insurance gate intendedWorkLot semantics
§5.4: optional intendedWorkLot field on the verdict request. Open question for Archidoc's AD2 plan: does it scope the verdict to lot-specific insurance requirements (e.g. some lots need décennale, others don't), or is it advisory-only audit metadata? If lot-scoped, Architrak's AT3 must always pass the lot when identifiable; if advisory, AT3 can defer. Proposal: Archidoc's AD2 plan calls this out before AT3 finalises.

§B.10 eventId UUID namespaces — fixture-reuse risk in shared test environments
Both Architrak's webhook_events (inbound dedup) and Archidoc's webhook_events_in (inbound dedup) key on eventId. Archisign's AS3 generates a fresh UUID per event; Architrak's outbound to Archidoc generates separately. Distinct namespaces in production. Risk in shared test envs: developers reuse UUIDs across replays, hit dedup, see "event silently dropped." Proposal: all three test plans use a UUID-per-test-run convention (e.g. UUIDv7 with timestamp prefix, or test-run prefix on a deterministic suffix) so dedup behaves correctly across replays.
