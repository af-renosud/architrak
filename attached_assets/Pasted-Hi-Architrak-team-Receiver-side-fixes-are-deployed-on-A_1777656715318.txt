1. What we shipped
Single transaction wrap (your request a): The dedup-insert into webhook_events_in and the entity-insert into work_authorisations (or signed_pdf_retention_breaches) now run inside a single db.transaction(...). A failed entity write rolls back the dedup row, so a subsequent retry from your orchestrator with the same eventId is not false-deduplicated — it goes through the full pipeline. This honors §1.4: dedup is committed only when the entity persists.

FK violation classification (your request b): SQLSTATE 23503 (foreign-key violation) is now mapped to HTTP 422 with a structured response body:

{
  "error": "Payload references an entity unknown to this server",
  "sqlstate": "23503",
  "hint": "Verify projectId / archidocProjectId / contractorId resolves on the receiver side"
}

Per §1.5 your orchestrator should dead-letter on 422 rather than feed the retry sweeper.

Bonus (caused by the same root cause): validation moved before dedup. Zod-rejected payloads now return 400 without touching the dedup ledger. This was the actual cause of the 2026-04-26 stranded rows — see §2 below.

Logger enriched. The previous [Architrak Work-Auth] Handler error: <drizzle failed-query summary> line swallowed SQLSTATE entirely (which is why our 19:00 CEST diagnosis required a manual schema dive). It now emits:

{
  "eventId": "...",
  "eventType": "...",
  "sqlstate": "23503",
  "message": "...",
  "stack": "..."
}

Phase-1 lookup failures (pre-transaction) get the same structured log line, just tagged Phase-1 lookup error (pre-transaction).
