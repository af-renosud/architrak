B. Integration-point gaps Archidoc spotted (input for Architrak's gap-spotting pass)
Seven items, ordered by criticality. None are contract-breaking; all are coordination details the contract doesn't explicitly nail down.

Test-fixture symmetry across apps. Contract specifies wire shapes (§5) but not shared test fixtures. If Archidoc T4's tests and Architrak's outbound-webhook tests use different example payloads with independently-computed HMACs, end-to-end verification across env boundaries is harder than it needs to be. Suggest a shared wire-fixtures/ convention in Architrak's cross-check artefact (purely operational, not a contract amendment).

ARCHITRAK_WEBHOOK_SECRET initial provisioning. §4 documents holder, rotation policy, and the 2-week dual-secret window — but not the initial secret-generation ceremony or the channel for first-time exchange. Architrak's plan likely covers this on their side; Archidoc needs the secret in our store before T4 can be smoke-tested in any env.

ARCHITRAK_API_KEY analogue. Architrak makes bearer-authed calls into Archidoc T3 via ARCHIDOC_SYNC_API_KEY — that's covered. There's no symmetric Architrak-issued API key for Archidoc → Architrak calls because Archidoc has no outbound calls into Architrak today. If Architrak's breakdown plans any new Archidoc-callable endpoint (e.g. a status-query endpoint for the architect to see "did Architrak receive my work authorisation?"), a new secret would need to enter §4. Worth confirming Architrak doesn't introduce one without contract amendment.

5xx response code discipline at the insurance-verdict endpoint. §1.3 says "503 or timeout >5s → red overridable". Archidoc T3 returns specifically 503 on transient errors. Architrak's gate logic should treat any 5xx (502/503/504, plus network timeouts) as the same overridable-with-warning UX — otherwise an intervening load-balancer or proxy returning 502 would be misclassified. Both readings are consistent with the contract; just worth alignment in Architrak's gap-spotting output.

eventType discriminator default semantics. Archidoc T4 defaults absent eventType to "work_authorised" per §5.3.2 backward-compat note. Architrak's outbound task should either always emit eventType (preferred — explicit beats implicit) or explicitly note reliance on the Archidoc-side default. Either is contract-conformant; the coordination is which convention both sides commit to in v1.0 production.

webhook_events_in retention/cleanup policy. §0.4 references "all three apps' existing webhook_events_in (or equivalent) idempotency tables" but Archidoc T1 creates this table net-new. The contract doesn't specify retention SLA, cleanup cadence, or maximum-row-age semantics for dedup. Long-term operational concern — propose v1.1 amendment candidate, not v1.0 blocker.

"Architect-of-record" notification on breach. T4's signed_pdf_retention_breach branch optionally emails the architect-of-record. Architrak's project model has a clear single owner; Archidoc's projects support multiple owners/collaborators. T4 currently ships with that gate hard-defaulted off pending the per-tenant setting. If Architrak's cross-check assumes Archidoc surfaces the breach via email automatically, that assumption needs flagging — banner-only is the v1.0 default behaviour.
